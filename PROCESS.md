## 돌 수집 — 내러티브 메모리 기반

- **소스**: mirror-mind/memory/narratives/ (세션 + 주제)
- **자동 생성**: close-session.py 실행 시 내러티브 메모리가 생성된다. 이것이 곧 돌이다
- **단위**: 세션 내러티브의 event 1개 = 돌 1개, 주제의 thesis/question/changelog = 돌 1개
- **인덱싱**: `python3 scripts/index-stones.py` (내러티브 → 임베딩)
- **검색**: `python3 scripts/find-stones.py --query "키워드"`
- **stones/ 폴더**: 인덱스(index.json, embeddings.npy)만 저장. 원본은 mirror-mind에 있다

## 소재 선정 기준

### 수집 단계 (넓게)
- 흥미로운 관찰, 숫자, 일화, 인용이면 일단 돌로 저장
- 아직 결론이 안 나도 된다. 나중에 다른 돌과 조합할 수 있다

### 발행 단계 (좁게)
1. 구체적 행동 + 숫자가 있다
2. IT 동료가 자기 경험에 대입 가능하다
3. 교훈이 이 프로젝트를 넘어서 적용 가능하다
4. 내부 프로젝트 배경 없이도 읽힌다
5. 실제 성과/결과가 나온 것이다

## 글쓰기 기법

제럴드 와인버그의 자연석(Fieldstone) 기법.
돌을 모으고, 글을 쓸 때 맞는 돌을 골라 조합한다.

## 글 구조 — 하이브리드 (돌 나열 + 시간순)

```
[돌로 시작] — 구체적 숫자/관찰로 훅
    ↓
[시간순 서사] — 전환점 중심으로 이야기 전개
    ↓
[돌로 마무리] — 패턴/교훈으로 수렴
```

## 글 규칙

- 태그: `경험` / `배움` / `생각` / `주간회고`
- 톤: ~다 체, 동료 수준, 솔직
- 길이: 400~800자 (한국어 기준)
- 프로젝트명: 일반화 (내부명 → "우리 AI 제품" 등)
- 한영: 한국어 먼저 → 영어 번역
- **사실은 기록에서만**: 숫자, 일화, 경험은 반드시 실제 기록(세션 서머리, 커밋, 코드)에서 가져온다. 지어내지 않는다. 출처를 추적할 수 있어야 한다

## 파이프라인

```
stones/  →  drafts/  →  content/ko/posts/  →  push (발행)
 수집        조합         완성                  배포
```

| 단계 | 폴더 | 상태 |
|------|------|------|
| 수집 | `stones/` | 돌 파일 |
| 초안 | `drafts/` | 하이브리드 구조로 조합 |
| 발행 | `content/ko/posts/` + `content/en/posts/` | Hugo 빌드 대상 |
| 배포 | GitHub push → Actions → Pages | 자동 |

## 인사이트 연결

- 인사이트는 독립적이지 않다. 하나의 경험이 여러 인사이트에 걸칠 수 있다
- 글을 쓸 때 다른 인사이트의 돌을 자연스럽게 가져다 쓸 수 있다
- 예: "계산+LLM" 에세이에서 "스펙이 안 맞아서 코드를 걷어냈다"는 돌을 끌어오는 식

## 외부 아이디어 접목

- 초안 작성 중 관련 외부 개념/프레임워크/사례가 있으면 웹 검색으로 원문을 확인한다
- 출처를 명시하고, 자기 경험과 대조하여 접목한다 (그대로 인용이 아니라 경험에 비춘 해석)
- 외부 아이디어는 기존 기록을 대체하지 않는다. 기록이 뼈대, 외부가 양념

## 대규모 작업 방식

소재가 N개일 때:
1. 소스 스캔 → 돌 추출 (서브에이전트 병렬)
2. 돌 → 선정 기준 필터
3. 통과한 소재별로 하이브리드 초안 작성 (서브에이전트 병렬)
4. 재영 검토 → 수정 → 발행
