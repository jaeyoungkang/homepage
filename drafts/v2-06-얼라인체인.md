에이전트가 532개 파일을 바꾸고 37,976줄을 추가했다. diff를 열었을 때 든 생각은 "이게 맞나?"였다. 코드를 한 줄씩 읽어서 검증하는 건 불가능하다. 그래서 다른 방법을 찾았다. 유저스토리에서 코드까지 선을 긋는 거다.

이걸 얼라인 체인이라고 부르기 시작했다. UserStory → Spec → Test → Code. 각 층이 아래층과 명시적으로 연결되어 있으면, 변경이 아무리 커도 스펙에서 출발해서 추적할 수 있다.

실제로 해본 작업이 있다. 뷰모드, 문서 한도, 히스토리 기능의 얼라인 체인을 완성하는 거였다. user-stories.md에 US-WORKSPACE-01-01부터 03-02까지 추가하고(+110줄), workspace-layout.spec.md를 신설했다(103줄). 거기에 vitest 5개 스위트를 새로 만들거나 갱신했다. 12개 파일, +788줄 -11줄. 숫자가 중요한 게 아니라, 유저스토리 하나를 고르면 그게 어떤 스펙 조건으로 내려가고, 어떤 테스트가 그걸 검증하는지 선이 보인다는 게 핵심이다.

feature-specs.md에는 워크스페이스 성공 기준과 수용 조건을 원장처럼 관리했다. SC-WORKSPACE 8개, AC 원장에 US-WORKSPACE 30행을 추가했다. 유저스토리 번호로 검색하면 스펙 조건이 나오고, 스펙 조건으로 검색하면 테스트가 나온다. 양방향 추적이 가능해진 거다.

물론 얼라인 체인을 만들었다고 끝이 아니다. 에이전트가 만든 코드가 실제로 체인과 맞는지 검증해야 한다. Codex를 4회 반복 리뷰하고, Claude로 구조 리뷰를 따로 돌려서 결함 15건을 잡았다. chip 클릭이 system event로 잘못 분류되어 있던 걸 user message로 전환한 것, interaction_events의 RLS를 강화한 것 같은 건 사람 눈으로는 놓치기 쉬운 것들이다.

코딩 에이전트의 산출물이 폭발적이라서 두렵다는 얘기를 종종 듣는다. 나도 그랬다. 그런데 US에서 코드까지 선을 그어보니까, 두려움이 아니라 체크리스트가 됐다. 선이 끊어진 곳이 버그고, 선이 없는 곳이 기술 부채다.
